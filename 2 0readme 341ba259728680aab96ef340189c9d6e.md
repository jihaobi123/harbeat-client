## 一、基于当前代码的完成度判断

### 1. 已完成且比较扎实的部分

### A. 两曲转场规划：完成度高

这个部分现在是项目最成熟的。

你现在的 `TransitionPlanner` 已经不只是简单看 BPM/key，而是做了多维打分：


而且候选搜索也不是单点判断，是在枚举：

- exit / entry bar
- overlap beats
- target bpm
- phase offset
- handoff profile
- transition strategy

这一层已经明显超过“玩具级别”。

### B. 多曲排序：已打通，但仍偏 heuristic

现在 `PlaylistPlanner` 已经能：

- 预计算 pair plan
- 小规模全排列找最优
- 大规模贪心排
- 加了一点 energy progression 和 strategy repeat penalty

```
    def _best_order(self, tracks: list[TrackMetadata], pair_plans: dict[tuple[str, str], TransitionPlan]) -> list[TrackMetadata]:
        if len(tracks) <= 7:
            return max(permutations(tracks), key=lambda order: self._sequence_score(list(order), pair_plans))

        start = min(tracks, key=lambda track: (self._intro_bias(track), self._avg_energy(track), track.title.lower()))
        ordered = [start]
        remaining = {track.track_id: track for track in tracks if track.track_id != start.track_id}
        while remaining:
            current = ordered[-1]
            next_track = max(remaining.values(), key=lambda candidate: self._pair_score(current, candidate, pair_plans))
            ordered.append(next_track)
            remaining.pop(next_track.track_id)
        return ordered
```

这说明它已经不是“随机排歌”了，是真有 DJ-aware heuristic 的。

但它还是没到“全局 set 设计”。

### D. 候选验证和分析报告：完成度不错

## 二、还没完成的关键缺口

### 1. 全局排歌仍然不够像 DJ

虽然多曲排序有了，但本质还是：

- pair 分数高优先
- energy progression 奖励
- intro/outro bias
- repeat strategy penalty

这还不是真正的 set-level objective。

它更像“避免排得太差”，还不是“故意排得好”。

也就是说：

**当前排歌系统 = 会避免明显不顺**

**目标排歌系统 = 能主动设计一条 journey**

---

### 2. 混音渲染虽然升级了，但还没到“专业自动混音”

虽然现在不是简化 crossfade 了，但仍然有几个明显现实差距：

- time stretch provider 还是以 `librosa` 为主
- 没有真实 deck state continuity 跨整场传递
- 没有更细的 EQ / phrasing automation 设计器
- 没有真实 transient-preserving / phase-robust 专业 sync 方案
- 还主要是 offline render，不是 performance-grade mixing

`sync.py` 里自己也把 `librosa` 定义成保守的 offline provider：

```
    if provider_name == "offline_hq":
        notes.append("offline_hq provider is more suitable for longer offline overlaps.")
        notes.append("offline_hq remains prototype-oriented and should stay offline-only.")
        return "hq", True, True, True, 32, 0.82, notes
    notes.append("librosa provider offers general-purpose offline time stretching.")
    notes.append("Long transitions should stay conservative with librosa provider.")
    return "standard", True, False, False, 16, 0.64, notes
```

所以这条线已经有骨架，但离“专业混音器”还远。

---

### 3. 实时引擎存在，但还没成为主产品能力

`GrooveAudioEngine` 已经在了：

```
class GrooveAudioEngine:
    """Coordinates decks, FX, and the output stream."""

    def __init__(self, sample_rate: int = 44100, block_size: int = 1024) -> None:
        self.sample_rate = sample_rate
        self.block_size = block_size
        self.command_queue: SimpleQueue[MixCommand] = SimpleQueue()
        self.deck_a = AudioDeck(deck_id="A")
        self.deck_b = AudioDeck(deck_id="B")
        self.fx = MixerFX(sample_rate=sample_rate)
        self.output_ring = RingBuffer(capacity_frames=sample_rate * 8, channels=2)
```

但目前 `web_app.py` 主链路里真正用于产出结果的是离线渲染，不是实时 engine。

所以实时 engine 现在更像：

- 技术储备
- 未来 performance 方向
- 不是当前必须推进的核心主线

---

### 4. 选歌能力基本还没开始

这个很明确。

现在系统的入口是：

- 先给候选 track_ids
- 再排序
- 再渲染

不是：

- 从大量歌库里挑哪些进 set
- 控制冗余
- 面向场景选最适合的歌

所以“选歌”现在确实还是后面的事情。

---

### 5. DJtransGAN 接入空间很大，但要注意接入点

如果你要尝试接 DJtransGAN，我建议你别把它当成“替代整个系统”的大脑。

更合理的是把它接成**一个额外评分器 / reranker / embedding provider**，而不是直接替换现有 rule-based planner。

因为你现在已有系统已经有：

- 明确可控的 transition grammar
- 可解释的多维分数
- 可渲染验证的链路

这很适合把模型接到：

- style compatibility score
- set stage score
- next-track suitability score
- candidate rerank
- transition strategy prior

而不是把整个 pipeline 推翻。

# 2. 排歌逻辑 / set 结构规划

### 当前完成度：**45%～55%**

### 当前优先级：**P0**

### 是否适合独立开发：**非常适合**

### 已有基础

- pairwise transition score 很完整
- playlist planner 已打通多曲排序
- 小规模 exhaustive / 大规模 greedy 都有
- 有一点 energy progression 和 transition diversity 约束

### 核心缺口

- 没有 set-level objective
- 没有 stage target（opening/build/peak/reset/close）
- 没有 context-aware next-track planning
- 没有“这首歌放在第 N 首是否合适”的位置分
- 没有 diversity / redundancy / pacing 的全局控制

### 为什么也是 P0

因为“像不像 DJ”最核心还是这条。

混音决定“好不好听”，排歌决定“像不像人在讲一段 set”。

### 建议负责人工作目标

这个人要把现在的 `PlaylistPlanner` 从“启发式排序器”升级成“整场结构规划器”。

### 这个人的近期任务顺序

1. 定义 set stages
    - opening
    - warm-up
    - build
    - peak
    - reset
    - close
2. 定义全局目标函数
    - energy curve
    - style coherence / contrast budget
    - strategy diversity
    - peak timing penalty
    - weak transition penalty
3. 替换当前排序搜索
    - beam search 优先
    - 不建议一上来就搞很复杂 RL
4. 增加可配置模式
    - smooth
    - hype
    - battle
    - reset-heavy
    - showcase

# 3. DJtransGAN / 模型接入

### 当前完成度：**10%～20%**

### 当前优先级：**P1**

### 是否适合独立开发：**可以独立，但建议接口先对齐**

### 为什么是 P1，不是 P0

不是因为它不重要，而是因为：

- 现在 rule-based 系统还没把 set objective 和 mix quality 打稳
- 模型现在接入太早，容易变成“加分但不稳”
- 如果底层 objective 不明确，模型接进去也会不知道该优化什么

### 适合你自己做的原因

你自己最清楚你想让模型学什么审美目标。

这块很适合你来做，但要注意接法。

### 最建议的接入方式

按独立模块拆：

1. **track embedding / style embedding**
2. **pair compatibility score**
3. **next-track suitability score**
4. **set-stage suitability score**
5. **reranker**

也就是先做成：

- 输入：track A、track B、当前 set context
- 输出：模型分数
- 再和 rule-based score 融合

而不是直接端到端替换。

### 建议优先接入的位置

先接排歌，不要先接混音。

优先顺序：

1. next-track rerank
2. set-stage score
3. transition strategy prior
4. candidate filtering
5. track selection
